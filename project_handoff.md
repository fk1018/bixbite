# PROJECT HANDOFF: Bixbite (.bixb) → Ruby (.rb) compiler (Rust) generating Sorbet sigs

## Goal

Build **Bixbite**, a typed superset of Ruby that compiles `.bixb` source files into normal Ruby `.rb` files runnable on CRuby. Types exist at dev-time; compiled output is standard Ruby. Static typechecking is performed by Sorbet (`srb tc`) via generated `sig { ... }` blocks.

**Non-negotiable product intent:** this should be a real tool with real-world adoption potential. Do not design for a rewrite—start with a real tokenizer/parser and structured diagnostics.

---

## Core user story

- I write this in `src/foo.bixb`:
  ```ruby
  def add(x: Integer, y: Integer) -> Integer
    x + y
  end
  ```
- Compiler outputs this to `build/foo.rb`:

  ```ruby
  # GENERATED BY BIXBITE — DO NOT EDIT
  # Source: src/foo.bixb

  sig { params(x: Integer, y: Integer).returns(Integer) }
  def add(x, y)
    x + y
  end
  ```

- Then I can run:
  - `srb tc` (on generated Ruby)
  - `ruby build/foo.rb`

---

## Defaults and conventions

- **Source of truth:** `src/**/*.bixb`
- **Output directory:** `build/**/*.rb` (gitignored)
- **Config file:** optional `bixbite.toml` (preferred) or `bixbite.json` (acceptable)
  - Defaults if absent: `source_dir=src`, `out_dir=build`
- **Tool name / CLI:** `bixbite`

## Deliverables (MVP v0.1)

### 1) Rust CLI

Implement `bixbite` CLI with subcommands:

- `bixbite build`
  - finds `.bixb` under `source_dir`
  - compiles each to `.rb` under `out_dir` preserving relative path
  - writes only if output content changed
  - prints human-readable summary
  - exits nonzero on errors

- `bixbite check`
  - runs `bixbite build`
  - optionally runs `srb tc` (if enabled via flag/config)
  - v0.1: it’s OK to just run build + emit diagnostics; Sorbet integration can be a flag (`--sorbet`)

**Important:** all diagnostics must be available in both human and JSON forms:

- `--format human` (default)
- `--format json` (for LLMs/automation)

---

### 2) Real tokenizer + parser (small grammar, not regex)

Implement a minimal grammar that supports **single-line method signatures** with typed params and typed return:

Supported signature forms:

- `def name(a: Type, b: Type = default) -> ReturnType`
- `def self.name(...) -> ReturnType`
- require parentheses in v0.1:
  - `def ping() -> NilClass` (allowed)
  - `def ping -> NilClass` (not allowed in v0.1)

Types allowed (v0.1):

- Ruby constant paths only: `String`, `Integer`, `Foo::Bar`
- Special-case: allow `Boolean` in `.bixb` and rewrite to `T::Boolean` in generated sigs

Params allowed (v0.1):

- positional params with optional defaults:
  - `x: Integer`
  - `y: String = "a"`
- v0.1 **does not** need kwargs splats, `*rest`, `**kw`, `&blk`

Method names allowed:

- `foo`, `foo?`, `foo!`, `foo=`
- `self.foo`, `self.foo?`, etc.

---

### 3) Strict rule for typed methods (v0.1 clarity)

If a `def` has a `-> ReturnType`, then **every parameter must be typed**.

Example (should error):

- `def f(x: Integer, y) -> Integer`

Error should point at the offending param span and suggest “add `: Type`”.

---

### 4) Code generation (Ruby emitter)

Emit Ruby with:

- Generated-file header at top:
  - `# GENERATED BY BIXBITE — DO NOT EDIT`
  - `# Source: <relative .bixb path>`
- One `sig { ... }` line immediately preceding the `def` line
- The `def` line emitted **without type annotations** but preserving defaults
- Everything else in the file should pass through unchanged for v0.1 (except the type annotation tokens are removed from the `def` line)

**Determinism requirement:**  
Same input must produce identical output every time.

---

### 5) Tests

Use Rust tests (`cargo test`). Provide a fixture-based test suite that validates:

- compilation output for a set of input `.bixb`
- error diagnostics (human and json) for invalid inputs
- “write only if changed” logic (optional but nice)

**Testing matrix:**

1. `def add(x: Integer, y: Integer) -> Integer`
2. `def greet(name: String, loud: Boolean = false) -> String` (Boolean rewrite)
3. `def self.encode(id: Integer, data: String = "00") -> String`
4. `def ok?() -> TrueClass`
5. `def set_name=(name: String) -> String`
6. whitespace variations in signature formatting

## Diagnostics format (required)

Diagnostics must include:

- file path
- message
- code (e.g., `BIX001`)
- severity (error/warn)
- span range (start line/col, end line/col)
- optional fix suggestion (can be simple text v0.1)

JSON example shape:

```json
{
  "diagnostics": [
    {
      "code": "BIX001",
      "severity": "error",
      "file": "src/foo.bixb",
      "message": "Typed method signature requires all params to be typed.",
      "range": {
        "start": { "line": 3, "col": 18 },
        "end": { "line": 3, "col": 19 }
      }
    }
  ]
}
```

## Non-goals (explicitly OUT for v0.1)

- Full Ruby grammar
- Multiline method signatures
- Union types, generics validation/inference
- Rails integration
- Runtime type checks
- Full LSP (but design diagnostics/spans so LSP is straightforward later)
- Translating Sorbet diagnostics back to `.bixb`

---

## Stretch goals (v0.2+)

- `Any` escape hatch, union types, generics
- kwargs/splats/&blk in signatures
- `bixbite lsp` (language server)
- `bixbite check --sorbet` runs `srb tc` and (later) maps errors to `.bixb`
